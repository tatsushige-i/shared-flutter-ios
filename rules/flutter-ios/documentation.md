# Documentation Rules

`docs/` 配下のドキュメント構成に関するプロジェクト固有の規範。docs/ 配下の
ファイルを追加・移動・削除する際は本ルールに従うこと。

## Directory Layout

```text
docs/
├── spec/      # 現行仕様（as-built）。実装の現状を正とし恒久的にメンテする
├── design/    # 未実装機能の設計案・検討資料
│   └── assets/    # 設計文書が参照する画像等
├── process/   # 開発・運用手順。現在有効な作業プロセスとしてメンテする
└── archive/   # 凍結済みの歴史的資料。更新しない
    └── <category>/  # live 構成をミラーしたカテゴリ別サブディレクトリ
```

## Directory Roles

| ディレクトリ | 役割                                       | ライフサイクル     |
| ------------ | ------------------------------------------ | ------------------ |
| `spec/`      | as-built の現行仕様書。実装に追従して更新  | 恒久メンテ         |
| `design/`    | 未実装機能の設計案・検討資料                | 実装完了まで       |
| `process/`   | 開発・運用手順。現在有効な作業プロセス      | 恒久メンテ         |
| `archive/`   | 役割を終えて凍結した文書（歴史的資料）      | 凍結（更新しない） |

## Placement Rules

- 新しいドキュメントは役割に応じて `spec/` / `design/` / `process/` のいずれかに置く
- ドキュメントが参照する画像等のアセットは、そのドキュメントと同じディレクトリ
  配下に `assets/` を作って置く（例: `design/assets/`）
- 階層は深くしすぎない。まず上記ディレクトリのいずれかの直下に置く

## Lifecycle

- `design/` の設計案は、実装完了時点で内容を `spec/`（現行仕様書）へ反映する
- 反映後、役割を終えた `design/` の文書は `archive/design/` へ移す
- `spec/` の文書が現行仕様でなくなった場合は `archive/spec/` へ移す
- `process/` の文書が現行プロセスでなくなった場合は `archive/process/` へ移す
- archive へ移す際はファイル移動のみとし、内容は凍結する（更新しない）

## archive/ Structure

- `archive/` 直下にファイルを直接置かず、live 構成をミラーしたカテゴリ別
  サブディレクトリ（`archive/spec/` / `archive/design/` / `archive/process/`）に置く
- 空のカテゴリディレクトリは作らない。中身が出た時点で作成する
